The Bench · № 01 · Apr 02, 2026

An audit log is the most underrated AI feature.

For a year we have been arguing about hallucinations, autonomy, and trust. The argument is interesting and mostly unresolved. Meanwhile, the simplest answer to most of those problems sits one design decision away: write down what the agent did.

An audit log is not a privacy feature. It is a sanity feature. When the agent reads a page, you should be able to read the page. When it writes a file, you should be able to find the file. When it spends three cents on a model run, you should see the three cents. The audit log is the workshop's clipboard — every job recorded, every tool returned to the wall, every piece of stock accounted for.

What ours looks like.

Append-only. Local. Replayable, which is to say: every entry has enough state to reconstruct the run that produced it. Exportable as JSON, which is to say: you can take it to a compliance review and read it without our help. Deletable, which is to say: when you delete the run, the log goes with it. There is no shadow copy on a server somewhere.

What we learned building it.

Three things. First, that an audit log changes how you write the agent — you start to design the behavior to read well in the log, which usually makes it better. Second, that users trust the system more once they know they can verify it, even if they never actually verify it. Third, that "log it locally and replay it later" obviates a lot of the elaborate observability infrastructure that cloud-first agentic systems end up bolting on.

Why we think every agentic system should have one.

Because debugging an agent without one is divination. Because security review of an agent without one is theatre. Because the right way to earn user trust is not to insist on it but to make the system show its work — and the audit log is the simplest possible way to do that.
